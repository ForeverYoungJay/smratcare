# 长者管理模块完整问题审计与重构方案

## 结论

当前长者管理模块已经不是单纯的功能缺失问题，而是典型的领域中枢缺失问题：

- 长者生命周期没有单一事实源
- 床位占用存在三份事实源
- 死亡和普通离院在主状态上混用
- 退住闭环此前断裂，现已补齐但边界仍不清晰
- 权限与审计此前依赖补丁式扩展

短期补丁已经覆盖高风险一致性问题，但要彻底解决，必须按领域重构。

## 已落地修复

- 修正 `resident-360` 路由入口错挂
- 对齐床位卡片状态语义
- 为长者/床位关键接口补显式 RBAC
- 为长者生命周期关键动作补结构化审计
- 修正普通外出状态回写
- 补齐退住结算后的自动闭环
- 补齐死亡登记释放床位和床位关系关闭
- 将统计口径中的“死亡”和“普通离院”拆开

## 根因拆解

### 1. 产品结构根因

- 缺少真正的“长者中心”
- 页面围绕功能菜单组织，而不是围绕长者生命周期组织
- 营销、评估、入住、在院、状态变更通过页面跳转串联，而不是通过领域中枢驱动

### 2. 业务逻辑根因

- `elder.status` 过于简化，无法表达完整业务态
- 外出、试住、死亡、退住申请、退住结算分别散落在多套表与流程
- 谁负责“最终离院闭环”边界不清，导致财务服务承担生命周期收尾

### 3. 数据结构根因

- `elder.bed_id`、`bed.elder_id`、`elder_bed_relation.active_flag` 并存
- `elder_change_log` 只能表示文本差异，不能承载完整事件语义
- 主状态未区分死亡离院和普通离院

### 4. 代码结构根因

- controller 过重
- 领域服务边界不清
- 统一枚举/状态帮助类不足，魔法值扩散
- 审计和业务留痕分裂成多套风格

### 5. 系统级风险根因

- 并发依赖代码流程控制，不依赖唯一事实源
- 新增业务时需要多点同步，极易回归
- 扩展“暂离院、失联、善后、审批撤回、清洁周转”等场景时复杂度会失控

## 最终重构目标

### 目标 1：建立统一长者生命周期状态机

建议拆为两层：

- 主状态：`INTENT / TRIAL / IN_HOSPITAL / OUTING / OUTING_MEDICAL / DISCHARGE_PENDING / DISCHARGED / DECEASED`
- 事件表：记录每一次外出、返院、入住、调床、退住、死亡、作废、审批

### 目标 2：建立唯一床位占用事实源

推荐以 `elder_bed_relation` 作为唯一当前占用来源：

- 当前占用：`active_flag = 1`
- 历史占用：`active_flag = 0`
- `elder.bed_id`、`bed.elder_id` 逐步降级为冗余缓存字段，最终只保留投影或移除

### 目标 3：统一审计模型

长者域关键动作统一写入结构化审计，包含：

- actor
- action
- entity
- before
- after
- context

### 目标 4：统一权限模型

将接口权限升级为“资源-动作”模型，而不是继续在 controller 上拼接角色字符串。

## 建议分阶段实施

### Phase 1：状态定义收口

- 建立统一状态常量/枚举
- 建立状态解释器和统计口径
- 清理前后端分散状态语义
- 抽出统一床位释放服务，收口退住、死亡、解绑床位三条链路

### Phase 1 当前落地结果

- 已新增统一状态常量：`ElderStatus`、`BedStatus`
- 已新增生命周期主状态字段与事件中心迁移：`V185__elder_lifecycle_state_and_event_center.sql`
- 已在 `elder` 主表补充 `lifecycle_status`、`departure_type`、`lifecycle_updated_at`、`last_lifecycle_event_id`
- 已新增 `elder_lifecycle_event`，开始记录生命周期状态迁移而不是只依赖 `elder_change_log`
- 已统一 `resident-360` 入口和床位卡片状态语义
- 已为长者、床位、生命周期关键接口补显式 RBAC
- 已将外出、试住、退住申请、死亡登记、床位解绑等关键动作补为结构化审计
- 已将试住登记正式接入 `TRIAL` 生命周期主状态，删除/转移试住记录时同步回写长者主状态
- 已抽出 `ElderOccupancyService`，统一处理床位释放、关系关闭、老人床位字段清理
- 已将床位解绑、死亡登记、出院办理、财务退住结算切到统一占床释放服务
- 已将入住办理、死亡登记、退住结算、财务退住状态同步切到统一 `ElderLifecycleStateService`
- 已将“外出”和“外出就医”从同一个 legacy `status=2` 中拆开，开始分别落到 `OUTING` / `MEDICAL_OUTING`
- 已将退住申请审核通过回写到 `DISCHARGE_PENDING` 主状态，不再只停留在申请表状态
- 已补充 `elder_bed_relation` 热点索引，降低关系查询和并发更新风险
- 已补充 `elder_bed_relation` 活跃占床唯一约束，防止同老人/同床位出现多条激活关系
- 已补充床位占用一致性巡检/修复接口，支持发现并修复老人、床位、关系表之间的投影撕裂
- 已将床位地图、床位列表、长者列表中的“当前床位/当前占床”解析改为激活关系优先，减少对 `elder.bed_id` / `bed.elder_id` 脏投影的依赖
- 已将财务工位中的住户分摊下拉、房间经营统计、退住状态同步校验切到“关系表优先、投影兜底”的读口径
- 已将长者列表、在院状态总览、长者中心入口切到以 `lifecycle_status` 为主的展示与筛选口径

### Phase 1 可上线边界

- 可以解决的核心问题：床位释放链路不一致、退住闭环断裂、死亡后仍占床、状态统计口径混乱、关键动作缺审计、核心写路径继续绕过生命周期中心
- 仍保留的结构性问题：仍存在少量 legacy `elder.status` 兼容读口径，`elder_bed_relation` 虽已成为主读写口径但 `elder.bed_id` 与 `bed.elder_id` 仍属于缓存投影而非正式移除

### Phase 1 上线前检查

- 先执行数据库迁移：`V183__elder_relation_guard_indexes.sql`、`V184__elder_bed_relation_active_unique_guard.sql`、`V185__elder_lifecycle_state_and_event_center.sql`
- 回归链路：入住分床、普通外出、外出就医、返院、死亡登记、退住申请、退住结算、手工解绑床位
- 重点核对：床位地图、长者列表、在院总览、退住单、死亡登记台账、变更日志、生命周期事件台账
- 核对审计：关键动作是否生成结构化审计与 `elder_change_log`
- 核对状态：`elder.lifecycle_status` / `departure_type` 是否与 legacy `status` 保持兼容映射
- 核对并发：同一老人重复退住申请、重复释放床位、结算后二次回写是否幂等、多人同时调床时是否被唯一约束正确拦截
- 核对巡检：`/api/bed/occupancy/health` 是否能发现历史撕裂数据，`/api/bed/occupancy/repair` 修复后是否恢复一致

### Phase 1 回滚策略

- 如仅业务逻辑异常，可先关闭相关入口并保留索引迁移不回滚
- 如需代码回滚，应整体回滚 `ElderOccupancyService` 接入链路，而不是单独回滚某一个 controller
- 索引迁移回滚应单独评估数据库窗口，不建议和业务回滚捆绑执行

### Phase 2：床位占用主表收口

- 将 `elder_bed_relation` 明确为唯一占用事实源
- 所有入住/调床/退住/死亡只操作关系表，再异步或同步刷新投影字段
- 增加唯一索引和并发约束

### Phase 3：生命周期事件中心

- 为入住、返院、外出、死亡、退住建立统一事件模型
- 将控制器内状态迁移逻辑下沉到领域服务
- 当前进度：主表字段与事件表已落地，核心写路径已开始接入，剩余工作主要是清理 legacy 读口径与补齐前端生命周期展示

### Phase 4：接口与页面重组

- 构建真正的“长者中心”
- 所有页面围绕单个长者生命周期展开
- 从“功能菜单”切换到“生命周期导航”

## 风险提示

- 如果直接大改表结构而不做兼容层，会造成前端和旧接口同步崩溃
- 如果只继续修 controller，不做领域重构，复杂度会继续指数级上升
- 如果不解决床位三份事实源问题，后续所有修复都只是补丁
