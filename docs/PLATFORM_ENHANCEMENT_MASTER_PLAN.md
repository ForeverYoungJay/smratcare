# 智养云（SmartCare）能力补齐总体方案

> 版本：v1.0　编制日期：2026-07-03
> 目标：在现有"机构一体化管理"底座之上，补齐"智慧养老"六大纵深能力，形成可招投标、可对接监管、可运营决策的完整平台。
> 交付方式：分期落地。本文件为**总体方案**，各期另有详细落地文档（一期见 `LTCI_REGULATORY_PHASE1_PLAN.md`）。

---

## 1. 现状盘点

现有平台已覆盖：长者全生命周期、护理、健康/体征、评估（含 AI 中医体质/心血管）、膳食、医养工作台、HR/OA、财务账单、资产物资库存、CRM 营销、商城、消防、家属小程序、员工移动端、RBAC 权限、微信支付/通知。技术栈为 Java 17 / Spring Boot 3.2.6 / MyBatis-Plus / Flyway / MySQL 8 / Redis，前端 Vue 3 + Ant Design Vue，微信小程序合并了家属端与员工端。

**当前 Flyway 基线：`V214`。新增迁移从 `V215` 起，按领域分段预留版本号（见 §4）。**

工程约定（新增代码须遵循）：

- 包结构：`com.zhiyangyun.care.<module>.{controller,service,service.impl,entity,mapper,model}`
- 接口前缀：管理端 `/api/<module>/...`，家属端 `/api/family/...`，统计 `/stats/...`，返回统一 `Result<T>`（`code=0` 成功）。
- 鉴权：`@PreAuthorize("hasAnyRole(...)")`，多租户通过 `AuthContext.getOrgId()` 过滤。
- 表约定：`id BIGINT` 主键（应用生成）、`tenant_id`、`org_id`、`create_time`、`update_time`、`is_deleted`，唯一键含 `is_deleted` 做软删。
- 角色模型：`SYS_ADMIN`/`DIRECTOR`/六部门 `*_MINISTER`、`*_EMPLOYEE`（MEDICAL/NURSING/FINANCE/LOGISTICS/MARKETING/HR），兼容 `ADMIN`/`STAFF`。

## 3. 各领域方案要点

### A. 长护险与监管对接（一期，最高优先级）

依据《长期护理失能等级评估标准（试行）》（国家医保局办公室、民政部办公厅，2021-08-03）与《长期护理保险失能等级评估管理办法（试行）》（国家医保局、财政部，2023-12）。

- **失能等级评估**：内置评估量表 = 3 个一级指标（日常生活活动能力、认知能力、感知觉与沟通能力）+ 17 个二级指标，采用组合法判定失能等级 0–5（0 能力完好 / 1 轻度 / 2–3 中度 / 4–5 重度）。支持初评、复评、争议复核流程与评估员、评估机构管理。
- **长护险待遇与结算**：参保登记、待遇资格、护理服务包与限额、按月结算与费用分摊（统筹支付/个人自付），生成结算清单。
- **民政上报**：对接"金民工程"全国养老服务信息系统，机构基本信息、床位、入住长者、服务记录按季度/事件上报，含上报任务、报文快照、回执与重试。
- **医保对接**：预留医保结算清单/电子凭证接口适配层（各地医保平台差异大，做适配器模式）。

> 详细数据模型、接口、Flyway、前端、验收见一期文档。**Flyway 预留 `V215–V229`。**

### B. IoT 安全预警闭环（二期）

在现有 `smart_device / smart_device_event / smart_alert` 上扩展：

- **场景规则引擎**：跌倒、SOS 一键呼叫、离床超时、离院（防走失）、卫生间/房间久滞、心率/血氧/睡眠异常，规则可按长者失能等级/护理等级配置阈值。
- **设备健康监控**：心跳超时→设备离线告警，防止"哑设备"漏报；电量、信号、固件版本巡检。
- **告警派单闭环**：触发 → 分级（危急/紧急/提醒）→ 自动派单到值班护理员移动端 → 响应/到场/处置/复盘，与 `incident`（不良事件）与护理工单打通，超时未响应自动升级。
- **视频/门磁/定位联动**：告警关联最近影像片段或定位轨迹（预留厂商适配层）。

Flyway 预留 `V230–V239`；接口前缀 `/api/smart/rules`、`/api/smart/dispatch`。

### C. 医养结合深度（三期）

- **电子病历（EMR）**：入院记录、病程记录、诊断、体格检查，关联现有健康档案与体征。
- **医嘱与处方**：长期/临时医嘱、执行单（给药、护理、检查），处方与药品目录关联。
- **药事管理**：药品目录、批号效期、库存（复用 `logistics/inventory` 或独立药房库）、发药/退药、高危药品与麻精药品管控、临期预警。
- **巡诊与随访**：医生巡诊排班与记录、慢病（高血压/糖尿病）随访计划与提醒。
- **急救对接**：一键急救流程、120 转诊记录、抢救记录单，与 IoT SOS 告警联动。

Flyway 预留 `V240–V254`；接口前缀 `/api/medical/emr`、`/api/medical/order`、`/api/pharmacy`。

### D. 经营驾驶舱 BI（四期）

- **经营指标域**：入住率、床位周转率、平均住院日、护理等级分布、护理工时/人效、营收/回款/欠费、成本结构、家属满意度、告警响应达标率。
- **领导驾驶舱大屏**：机构/连锁两级视图，支持按园区、时间维度下钻。
- **指标口径治理**：统一指标定义与计算口径，离线预聚合（定时任务写 `stats_*` 汇总表）+ 实时补数，避免大屏直查业务库。

Flyway 预留 `V255–V262`；接口前缀 `/stats/executive`、前端新增 `views/cockpit`。

### E. AI 提效（五期）

- **智能排班**：结合护理等级、工时法规、资质约束、休假，求解值班表，人工可调整。
- **健康风险预测预警**：基于体征/评估/用药趋势的规则+模型预警（跌倒风险、压疮风险、再入院风险）。
- **异常识别**：接入 IoT 事件做行为异常识别（久卧、夜间频繁离床）。
- **语音陪伴/智能客服**：家属答疑、长者语音交互（复用现有语音消息能力）。

以现有 AI 评估服务为底座扩展，模型能力走可插拔的 AI 网关适配层，Flyway 预留 `V263–V270`。

### F. 合规与数据安全（贯穿各期，六期收口）

- **等保 2.0**：三级等保要求的日志留存（≥6 个月）、双因子、会话与密码策略。
- **个人信息保护**：老人健康/家属隐私为敏感个人信息，落地分级分类、最小权限、脱敏展示（手机号/身份证/病情）。
- **访问审计**：在现有 `audit` 上补齐"谁在何时查看/导出了哪位长者的敏感数据"，导出行为二次确认+留痕。
- **备份容灾**：数据库定期备份与恢复演练、关键配置密钥托管（生产 JWT/支付密钥禁用默认值）。

Flyway 预留 `V271–V276`。

## 5. 分期路线图（Now / Next / Later）

**Now（一期，约 6–8 周）— A 长护险与监管对接**
失能评估 → 长护险待遇/结算 → 民政上报 → 医保适配层。产出可演示、可对接试点城市。

**Next（二、三期，约 10–14 周）— B IoT 安全预警 + C 医养结合**
先做 B 拿"智慧"卖点与安全闭环；C 补医养资质与临床安全，两者在急救/告警处联动。

**Later（四、五、六期）— D 经营 BI + E AI 提效 + F 合规收口**
D 支撑连锁扩张决策；E 在数据积累后价值最大；F 贯穿实施、上线前专项验收。

每期统一节奏：数据模型 → Flyway → 后端接口 → 前端页面 → 权限点 → 回归脚本 → 验收清单。

## 6. 横切关注点

- **多租户/机构隔离**：所有新表带 `org_id`，查询走 `AuthContext.getOrgId()`。
- **权限点**：新模块沿用六部门角色；监管/医保相关操作建议新增细粒度权限点（如 `LTCI_ASSESS`、`GOV_REPORT`）。
- **对接适配层**：民政、医保、IoT 厂商、AI 模型均走"适配器 + 配置"模式，避免与具体供应商强耦合，便于跨地区/跨厂商部署。
- **回归与自检**：每期新增对应 `scripts/*_smoke.sh` 与 Postman 集合，纳入现有 `project_self_check.sh`。

## 7. 风险与依赖

| 风险 | 应对 |
|------|------|
| 各地长护险/医保接口标准不统一 | 适配器模式 + 配置化，先支持试点城市，接口留扩展点 |
| 民政上报字段随政策变动 | 报文模板与字段映射配置化，保存报文快照便于追溯 |
| IoT 厂商协议差异大 | 统一事件模型 + 厂商网关适配层，设备侧只需上报标准事件 |
| 敏感数据合规 | F 领域贯穿实施，上线前做等保与个保专项验收 |
| 版本并行冲突 | Flyway 分段预留 + 提交前版本冲突检查 |
